# AI Planning Copilot for Shop Floor Scheduling

FastAPI project for the A*Star take-home assignment, with a tiny plain HTML
test UI served by the API.

Current status: deterministic backend endpoints and the `/api/v1/ask` hybrid
router are implemented. The app can be containerised with PostgreSQL and
Ollama, query seeded scheduling data, run deterministic scheduling tools, use
DeepSeek-R1 for reasoning/router extraction, and use Qwen for read-only SQL
plus answer-support confidence judging.

## What Exists So Far

- FastAPI backend under `Backend/app/`
- PostgreSQL seed data in `Data/seed.sql`
- Docker Compose setup for:
  - `api` - FastAPI service
  - `db` - PostgreSQL database
  - `ollama` - local LLM service using `deepseek-r1:1.5b` and `qwen2.5:3b`
- Health endpoint:
  - `GET /api/v1/health`
- Deterministic scheduling endpoints:
  - `GET /api/v1/machines/load`
  - `GET /api/v1/orders/at-risk`
  - `POST /api/v1/simulate/downtime`
- GenAI endpoint:
  - `POST /api/v1/ask`
- Test UI:
  - `GET /ui/`
- Basic health tests in `tests/`

`/api/v1/ask` first extracts intent and entities, verifies IDs against
PostgreSQL, then routes questions to simple tools:

- `run_sql` - Qwen generates one safe PostgreSQL `SELECT` query.
- `check_load` - deterministic Python load calculation from machine and order rows.
- `get_priority` - deterministic Python priority filtering from live order rows.
- `simulate_downtime` - deterministic Python simulation after reading machine/order data.
- `recommend` - deterministic recommendations from load and risk data.
- `refuse` - out-of-scope questions.

Obvious IDs such as `WO-1003`, shorthand like `1003 current status`, and
machine IDs such as `M2` are parsed first. DeepSeek-R1 then provides visible
router reasoning in a `<think>` block and fills fuzzy entities such as
`machine beta`. Qwen is used for SQL generation in the `run_sql` path and for
a structured confidence-judge pass after the answer is built. Each
`/api/v1/ask` request writes a best-effort audit row to `agent_action_log`.

Responses split the interface into `answer`, `explanation`, `sql_used`, and
`data`. `answer` is the concise scheduling answer. `explanation` is the parsed
DeepSeek-R1 `<think>` reasoning used for routing/backtesting. `confidence` is
a judge-assessed support score from `0` to `1`; it is not a guarantee of
objective truth.

## Local LLM Model

Docker Compose pulls the configured models automatically through the
`ollama-pull` service:

```text
deepseek-r1:1.5b
qwen2.5:3b
```

Model roles:

- `deepseek-r1:1.5b` routes intent, extracts fuzzy entities, and exposes
  routing reasoning. It is a DeepSeek-R1 distilled Qwen model; DeepSeek
  describes the distill family as using reasoning data generated by DeepSeek-R1
  and curated distillation samples.
- `qwen2.5:3b` generates structured read-only SQL and judges whether the final
  answer is supported by the returned evidence. Qwen2.5 is instruction-tuned,
  strong at structured JSON-style outputs, and described by Qwen as pretrained
  on large-scale multilingual, code, and math data.

If the first startup is interrupted, this troubleshooting command is safe to run:

```powershell
docker exec -it scheduling_ollama ollama pull qwen2.5:3b
docker exec -it scheduling_ollama ollama pull deepseek-r1:1.5b
```

## Audit Logging

`/api/v1/ask` writes one audit row after each response is built. The log records:

- `session_id`
- `action_type`
- `input_question`
- `sql_generated`
- `result_summary`
- `confidence`

Logging is best-effort. If the audit insert fails, the API still returns the
normal response.

Inspect recent logs:

```powershell
docker exec -it scheduling_db psql -U postgres -d scheduling_db -c "SELECT action_type, input_question, sql_generated, result_summary, confidence, created_at FROM agent_action_log ORDER BY created_at DESC LIMIT 5;"
```

## Prompt Templates

Prompt templates live in:

```text
Backend/prompts/
```

- `.env.example` is the committed source of runtime config defaults:
  `DATABASE_URL`, `OLLAMA_BASE_URL`, model names, and LLM timeouts.
- Docker Compose loads `.env.example` directly with `env_file` for the API
  and Ollama model-pull helper.
- `Backend/app/config.py` is the Python-facing settings loader. It reads the
  runtime environment and only keeps safe local-dev fallbacks.
- `schema_context.yaml` contains table meanings, known values, views, and business rules.
- `router.yaml` contains DeepSeek-R1 routing, reasoning, and fuzzy entity examples.
- `sql_generation.yaml` contains Qwen SQL-generation role and safety instructions.
- `judge.yaml` contains Qwen answer-support scoring instructions.

The YAML files guide the local models only. SQL execution safety is still
enforced in Python by `db.is_safe_select()`.

Blueprint configuration notes:

- Prompt context node: loads `schema_context.yaml`; context only, no SQL enforcement.
- Router node: loads `router.yaml`; DeepSeek-R1 emits `<think>` reasoning plus structured intent/entities.
- SQL generation node: loads `sql_generation.yaml`; Qwen generates candidate `SELECT` SQL for the `run_sql` path only.
- Confidence judge node: loads `judge.yaml`; Qwen returns structured JSON with a support score, verdict, reason, and issues.
- SQL safety node: Python rejects non-SELECT SQL, destructive SQL, and multiple statements before execution.
- Entity validation node: Python checks extracted work-order and machine IDs against PostgreSQL before routing.
- Known values: enum/category-like values live in YAML; live operational records stay in PostgreSQL.

## Known Limitations

- The UI is intentionally minimal: plain HTML and browser JavaScript only.
- No multi-turn conversation memory.
- Tool selection is hybrid: deterministic parsing first, DeepSeek-R1 reasoning router for fuzzy wording, then DB validation.
- First Docker startup downloads Ollama, `deepseek-r1:1.5b`, and `qwen2.5:3b`, so it can take time.
